## 1. 概述

### 1.1 系统定位

ReToDoScheduler 是一个**自进化的外脑调度系统**。系统通过三层协作帮助用户管理待办事项，核心目标是将用户的认知负担外化为可信赖的确定性规则，同时借助 AI 持续提升调度智能。

### 1.2 核心飞轮

```
人 → AI 建议 → 人调整 → 规则沉淀 → 更好的 AI 建议 → …
```

### 1.3 三层协作模型

| 层 | 职责 | 实现特征 |
|---|------|---------|
| **确定性规则引擎** | 可预测、可审计、可导出的调度逻辑 | 纯文本规则，用户可完全掌控 |
| **AI 估算层** | 从自然语言和历史行为中提炼建议 | 外部 API，提供建议但不做最终决定 |
| **人类反馈环** | 用户调整被前两层吸收，驱动系统自我改造 | 统计规律发现 → 规则建议 → 用户审核 |

### 1.4 设计原则

- **Phase α 独立可用**：不依赖任何 AI，是完整可用的产品
- **渐进增强**：后续 AI 能力叠加不破坏已有功能
- **每个 AI 组件有手动回退路径**：AI 不替代手动，而是逐步自动化
- **纯文本可导出**：用户对数据拥有终极控制权

### 1.5 技术选型

| 层 | 技术 |
|----|------|
| Monorepo | npm workspaces |
| Web | React 18 + Vite 5 + TypeScript |
| Mobile | React Native 0.76 + Expo 52 + expo-sqlite |
| API | Fastify 5 + PostgreSQL 16 |
| 共享逻辑 | `@retodo/core` (packages/core) |
| 同步策略 | Local-first + LWW (Last-Writer-Wins) |

> 详见 [ADR-0001](adr/0001-monorepo-and-stack.md)、[ADR-0002](adr/0002-local-first-sync.md)

## 2. 系统架构

### 2.1 分层架构总览

```
┌─────────────────────────────────────────────────┐
│                  客户端层                         │
│  apps/web (React+Vite)  │  apps/mobile (RN+Expo) │
└────────────────┬────────┴───────┬────────────────┘
                 │   @retodo/core │
┌────────────────┴────────────────┴────────────────┐
│              共享领域逻辑层 (packages/core)        │
│  Norn (领域模型/解析/分类)  │  Kairos (调度/打分)  │
└────────────────┬────────────────────────────────┘
                 │  /v1/tasks/sync (LWW)
┌────────────────┴────────────────────────────────┐
│              服务层 (services/api, Fastify)       │
│  认证 (Bearer Token) │ 同步 │ 持久化             │
└────────────────┬────────────────────────────────┘
┌────────────────┴────────────────────────────────┐
│              存储层 (services/db, PostgreSQL 16)  │
└─────────────────────────────────────────────────┘
```

### 2.2 Norn / Kairos 职责划分

| 概念 | 命名来源 | 职责边界 |
|------|---------|---------|
| **Norn** (命运织者) | 北欧神话 | 定义待办事项如何被**记录、结构化、分类和关联** |
| **Kairos** (恰当时机) | 希腊哲学 | 定义系统如何帮用户决定**「现在该做什么」** |

### 2.3 Kairos 三层调度架构 (v0.3)

AI 与确定性规则的分工：**AI 做创造性判断（调度主力），确定性规则做刚性校验（兜底保障）**。

```
┌─────────────────────────────────────────────────┐
│               人类反馈环                          │
│  用户对 AI 建议的调整                             │
│  → AI 自动发现统计规律 → 规则建议 → 用户审核通过  │
│  → 纳入确定性规则                                │
└────────────────┬────────────────────────────────┘
                 ↓ 吸收
┌─────────────────────────────────────────────────┐
│      AI 调度层（外部 API，启发式，调度主力）       │
│  · 自然语言 → 结构化字段                          │
│  · 任务拆分 / 优先级排序 / 时间片分配              │
│  · 历史行为模式 → 校准                            │
│  触发时机：任务创建/编辑时 | 队列变动时            │
└────────────────┬────────────────────────────────┘
                 ↓ AI 输出
┌─────────────────────────────────────────────────┐
│     确定性校验层（纯文本可导出，刚性兜底）         │
│  · 不遗漏：所有应调度任务均已分配时间片            │
│  · 不差错：分割后耗时总和 = 原始估算               │
│  · DDL 守护：DDL 前有足够时间                     │
│  · 排序校验：结果符合设计约束                      │
│  · 原型规则：可见性行为（唤醒/温度递升/…）         │
│  · 学习到的校准参数                               │
└─────────────────────────────────────────────────┘
```

## 3. 领域模型 — Norn

### 3.1 价值光谱

所有任务分布在一条价值光谱上：

```
争得 ←————————————————————————————→ 避祸

"不做死不了，做了开启可能性"        "做了不改变现状，不做带来毁灭性后果"
 高收益 · 雪球效应 · 机会驱动        防御性 · 不可接受损失 · 存续驱动
```

每个任务有一个**固有位置**（在此光谱上），用户有一个**决策向量**，Kairos 依据 `任务位置 × 决策向量` 加权优先级排序。

#### 3.1.1 决策向量

决策向量是**可扩展的键值对结构**，初始为一维标量（争得↔避祸），随使用浮现新维度。新维度通过 AI 提议 → 用户命名确认后加入。

已识别的候选维度：

| # | 维度 | 说明 |
|---|------|------|
| 1 | 争得 ↔ 避祸 | 首要维度，已确认 |
| 2 | 干系人权重 | 来源人的重要性 |
| 3 | DDL 紧急度 | 截止日迫近程度 |
| 4 | 个人成长价值 | 技能/认知提升空间 |
| 5 | 难度/耗时/风险 | 执行门槛与不确定性 |
| 6 | 曝光/影响力 | 在关键人面前的可见性 |
| 7 | 可分粒度/可并行度 | 在时间片装箱问题中的灵活程度/能否与其他任务并行 |

### 3.2 任务原型

六种原型覆盖生活和工作两个领域。工作任务在同一套原型上叠加额外元数据。

| 原型 | 时间行为 | 调度决策权重 | 生活例 | 工作例 |
|------|---------|-------------|--------|--------|
| **时限义务** | 休眠 → 唤醒 | 中~高 | 换驾照 | 写 ToD 文档 (DDL 一周) |
| **随缘备忘** | 温度递升 → 审视 | 低 | 改与会人身份 | review 同事代码 |
| **周期仪式** | 周期触发 | 低 | 每周审视 | 逆向改进月报 |
| **修行课题** | 持续在场 + 检查点 | 高 | 品味工作日志 | 责任田交接 (多步骤) |
| **轻量种草** | 永久休眠 | 无 | 学金融 | — |
| **Techne（技艺）** | 持续在场 | 中→高 | 歌单同步工具 | 整理自测试规范 |

#### 分类轴

| 轴 | 取值 | 说明 |
|----|------|------|
| 时间行为 | 休眠→唤醒 / 永久休眠 / 周期触发 / 持续在场 | 决定任务何时对用户可见 |

### 3.3 工作任务元数据层

工作任务在原型之上需要额外元数据（生活任务可选填）：

| 元数据 | 说明 | 输入方式 |
|--------|------|---------|
| 干系人 | 谁关心此任务，关系动力学 | 从干系人知识图谱选择 |
| 阻塞关系 | 你阻塞谁 / 谁阻塞你 | 手动标注 |
| 绩效挂钩 | 影响哪个考评维度 | 从环境上下文选择 |
| 耗时范围 | 支持乐观/悲观双值 (如 0.5~2 天) | 手动或 AI 推断 |
| 信息不确定性 | 耗时是否取决于调查结果 (条件分支) | 手动标记 |

### 3.4 干系人知识图谱

采用**渐进式构建**，不做一次性录入，从日常任务输入中逐步积累：

| 层 | 构建方式 | 示例 |
|----|---------|------|
| **原始层** | 从任务描述中自动积累文本片段 | `"刘某，同组，职级略高，有求于他"` |
| **结构化层** | AI 渐进提炼，用户确认后生效 | `刘某: { 关系: 同组, 职级: 略高, 影响: 中等加成 }` |
| **主动补问** | AI 识别知识缺口后择机提问 | 创建任务后 / 审视仪式时 / 专设区域 |

干系人画像直接影响任务在价值光谱上的定位——如大 SE 相关任务即使本身价值低也要优先处理。

### 3.5 工作流模板

需求开发等有固定阶段的重复性工作，用**模板**承载。工作流不需要独立 UI，**复用修行课题的进展队列结构**。

```
模板示例: 需求开发（1~2 周）
  ① 设计文档
  ② 测试串讲
  ③ 开发
  ④ 自验证迭代
  ⑤ 质量测试
  ⑥ 上库流程（~1 周）
```

- **Phase α**：修行课题的进展队列同时承载工作流跟踪
- **Phase β+**：模板实例化，新需求一键创建含预填阶段的进展队列

Norn 概念层次：`Task → Workflow → Template`

### 3.6 输入规范

**唯一不可缺失的字段：标题。**

| 字段 | Phase α（无 AI） | Phase β+（有 AI） |
|------|----------------|-----------------|
| 标题 | **必需** | **必需** |
| 原型 | 必需（下拉选择） | AI 推断 → 人确认 |
| 截止日 | 有则填 | AI 从文本提取 |
| 耗时估算 | 有则填（支持范围值） | AI 推断 |
| 干系人 | 有则选 | AI 从文本匹配图谱 |
| 自然语言备注 | 可选 | 越多越好，供 AI 学习 |

Phase β+ 支持**动态输入提示**：系统根据上下文智能追问关键信息，用户可跳过。

最简输入示例：
> `提API申请，找刘某，本周` → 系统自动补全干系人关系、推断原型和价值位置

## 4. 调度引擎 — Kairos

### 4.1 优先级计算

优先级由以下因子加权计算：

| 因子 | 来源 | 权重特征 |
|------|------|---------|
| 价值光谱位置 × 决策向量 | Norn | 核心排序依据 |
| DDL 紧急度 | 任务截止日 | 越临近权重越高 |
| 任务可分割度 | 最小不可分割粒度 | 大块任务获得额外加成 |
| 干系人权重 | 知识图谱 | 关键干系人相关任务优先 |
| 精力水平 (未来) | 执行力数据建模 | Phase α 视为恒定 |

### 4.2 任务可分割度

任务有**最小不可分割粒度**，这是 Kairos 的装箱约束：

| 可分割度 | 最小块 | 调度特征 | 例子 |
|---------|--------|---------|------|
| 极细 | ~10 min | 随时做，填缝隙 | 回消息 |
| 中等 | ~1 hr | 需要连续时间段 | review 代码 |
| 粗 | ~2-4 hr | 需要半天整块 | 解 bug 走读 |
| 不可分割 | 半天+ | 必须完整大块 | 需求开发-开发阶段 |

**调度策略**：大块任务需要优先级加成——可用的大时间窗口稀缺，如果不优先安排会被碎片任务不断挤走。本质是**装箱问题：大块先放**。

### 4.3 时间域

工作任务和生活任务调度在不同时间域，紧急时允许互相侵占：

| 时间域 | 区间 | 主要任务类型 |
|--------|------|-------------|
| 工作时间域 | 工作日工作时段 | 工作任务 |
| 生活时间域 | 下班后 + 周末 | 生活任务 |

**溢出规则**：紧急 DDL 可侵占对方时间域。

### 4.4 条件分支耗时

工作任务常见「耗时取决于调查结论」的模式，系统支持：

| 字段 | 说明 |
|------|------|
| 乐观估算 | 最好情况的耗时 |
| 悲观估算 | 最坏情况的耗时 |
| 当前判断 | 待调查 / 偏乐观 / 偏悲观 |

调度时按**悲观估算**预留时间，实际进展后可修正。

### 4.5 输出形式

| 视图 | 定位 | 说明 |
|------|------|------|
| **时间块日历** (主视图) | 日视图 + 周视图 | 任务占据具体时间片 |
| **任务序列总表** (辅助视图) | 全量队列 | 所有任务按优先级排列 |

### 4.6 原型调度策略

#### 4.6.1 时限义务 — AI 解析流程

1. 用户输入自然语言描述
2. AI 解析输出：截止日、外部依赖、风险因子、建议唤醒日、建议执行窗口
3. 用户确认或修正
4. 修正结果回馈 AI 上下文，校准未来解析
5. 解析用的 prompt 模板为纯文本，可导出

#### 4.6.2 随缘备忘 — 温度递升

| 阶段 | 入库后时间 | 系统行为 |
|------|-----------|---------|
| 冷藏 | 0~7 天 | 完全隐藏，脑卸载 |
| 微温 | 7~30 天 | 每周在「轻推」区域出现 1 次 |
| 升温 | 30~90 天 | 每周出现 2~3 次，视觉权重提高 |
| 审视 | 90 天+ | 触发三选一：做掉 / 重置冷藏 / 无愧归档 |

阶段参数作为确定性规则，可导出可调。

#### 4.6.3 修行课题 — 进展队列

```yaml
课题: "<标题>"
总目标: "<描述>"
最终期限: <日期>
进展队列:
  - 日期: <更新日期>
    内容: "<进展描述>"
    下次检查点: <日期>
最后更新: <日期>
下次提醒: <日期>
```

#### 4.6.4 Techne（技艺） — 双态模型

> 命名来源：希腊语 τέχνη，亚里士多德的「能产出成果的实践知识」。

- **轻量态**：单条任务，AI 估算耗时，简单则直接完成归档
- **升级操作**：用户或 AI 判断为复杂 → 转化为修行课题，获得子项/进展跟踪能力

#### 4.6.5 周期仪式 — 归属决策 [待定]

| 子类型 | 建议归属 | 状态 |
|--------|---------|------|
| 审视仪式 (每周/每月审视) | 内建为 Kairos 节拍器机制 | 待确认 |
| 周期性事务 (月度结算) | 外部提醒工具 (iOS 提醒事项) | 待确认 |
| 工作周期仪式 (逆向改进月报) | 待确认 | 待确认 |

### 4.7 精力水平因子

**Phase α**：视为恒定，不作为调度输入。但从 Phase α 起即采集以下数据，为未来建模埋点：

| 采集指标 | 含义 | 未来用途 |
|----------|------|---------|
| 完成度 | 任务是否完成/部分完成 | 估算实际难度 vs 预估难度 |
| 实际耗时 vs 预估耗时 | 用户的估算偏差 | 校准耗时估算系数 |
| DDL 遵守度 | 是否在截止日前完成 | 识别哪类任务容易拖延 |

**未来 Phase**：基于积累数据建立精力波动模型，使调度考虑「什么时候适合做什么」。

## 5. 处理管道

### 5.1 任务输入管道

```
[用户输入]
  → [解析器]      α: 表单直填       β+: AI 自然语言解析
  → [增强器]      α: 无操作         γ+: AI 推断隐含字段
  → [用户审阅]   ← 永远保留
  → [存入 Norn]
```

### 5.2 调度管道

```
[Norn 任务] + [时间可用性] + [决策向量] + [精力水平]
  → [优先级评分器]   α: 确定性公式    γ+: 公式 + AI 加权
  → [时间片分配器]
  → [日历视图 (日/周)] + [任务序列总表]
```

### 5.3 反馈管道

```
[用户调整记录]
  → [模式检测器]   α: 禁用           β+: AI 分析
  → [规则建议]
  → [用户审核]   ← 永远保留
  → [确定性规则更新]
```

## 6. 数据设计

### 6.1 核心实体

| 实体 | 职责 | 关键字段 |
|------|------|---------|
| `Task` | 主任务 | id, title, rawInput, status, estimatedMinutes, minChunkMinutes, dueAt, importance, value, difficulty, postponability, taskTraits, tags, extJson |
| `TaskPart` | 子任务（拆分调度） | taskId, 继承 Task 的调度约束 |
| `TimeWindow` | 周期性可用时间模板 | 定义用户的可用时间框架 |
| `TimeSlot` | 具体可调度时间段 | id, startAt, endAt, slotTraits (focus/interruptibility/location/device) |
| `ScheduleBlock` | 任务到时间段的分配 | taskId, slotId, startAt, endAt, isParallel |
| `LearningEvent` | 拖拽排序偏好样本 | higherTaskId, lowerTaskId |
| `SyncOp` | 跨端增量同步事件 | 用于 LWW 同步协议 |

### 6.2 实体关系

```
Task 1──* TaskPart          (一个任务可拆分为多个子任务)
Task 1──* ScheduleBlock     (一个任务可占据多个时间块)
TimeSlot 1──* ScheduleBlock (一个时间段可承载多个任务块)
TimeWindow ──> TimeSlot     (模板生成具体时间段)
Task *──* Task              (阻塞关系: blocked_by / blocks)
Task *──1 Stakeholder       (干系人关联, 未来扩展)
```

### 6.3 TaskTraits 与 SlotTraits 匹配

任务特质 (`TaskTraits`) 与时间段特质 (`TimeSlotTraits`) 进行匹配，确保任务被分配到合适的时间段：

| 特质维度 | 说明 |
|---------|------|
| focus | 所需专注度：high / medium / low |
| interruptibility | 可打断性：low / medium / high |
| location | 场所约束：indoor / outdoor / any |
| device | 设备约束：desktop / mobile / any |

### 6.4 Norn 待扩展字段

以下字段需在 `Task` 实体上新增或在 `extJson` 中承载：

| 字段 | 类型 | 说明 |
|------|------|------|
| archetype | enum | 任务原型 (6 种) |
| domain | enum | 领域 (work / life) |
| spectrumPosition | number | 价值光谱位置 |

### 6.5 纯文本持久化

| 设计原则 | 说明 |
|---------|------|
| 用户信任 | 纯文本是唯一真相来源的可验证副本 |
| 人机双可读 | 导出文件必须人可阅读 |
| 数据量预估 | 较小，单文件导出可行 |

导出内容：Norn 数据（全量任务及关联数据）+ Kairos 规则（调度参数与确定性规则）。

## 7. 用户场景

### 7.1 Phase α 核心场景

#### UC-1: 认知卸载 — 任务录入

- **触发**：用户想到一件待办事项
- **操作**：通过 Web 端表单录入标题及可选元数据
- **预期**：任务入库，大脑释放认知负担
- **核心价值**：录入本身即价值，不依赖后续调度

#### UC-2: 承诺时间块 — 日历拖拽

- **触发**：用户决定开始工作
- **操作**：在日视图上拖拽任务到时间块
- **预期**：用户承诺的是**时间而非结果**，减轻完美主义焦虑
- **设计理念**：「承诺时间块，而不是承诺任务完成」

### 7.2 远期场景

#### UC-3: 快速捕获

- **触发**：正在做任务 A，突然想到任务 B
- **操作**：悬浮按钮/语音极速输入
- **预期**：不打断心流，任务 B 入库

#### UC-4: 每日启动简报

- **触发**：早上打开 app
- **操作**：查看 Kairos 生成的日计划
- **预期**：快速进入执行状态，抵御启动困难

#### UC-5: 跌落后回归

- **触发**：用户几天没使用系统
- **操作**：重新打开 app
- **预期**：系统不追责、不堆积罪恶感，提供「现在从这里开始」的切入点

#### UC-6: 审视仪式

- **触发**：Kairos 节拍器触发每周/每月审视
- **操作**：回顾完成情况、调整优先级、校准决策向量
- **预期**：近视（日常执行）与远视（战略校准）的节奏性交替

## 附录 A：设计决策记录

> 以下为产品设计过程中的关键决策，按主题归类。

### A.1 任务原型与分类

- 从 9 个实际待办样本识别出 6 种原型
- 分类轴确定为「时间行为 × 调度决策权重 × 领域」
- 10 个工作任务验证六种原型未被推翻，工作任务通过元数据层扩展

### A.2 调度架构演进

- 初始：确定性规则引擎 + AI 估算层并列
- v0.3：明确 AI 为调度主力，确定性规则为校验兜底
- 任务可分割度确认为关键调度维度（装箱问题）

### A.3 价值体系

- 「争得 ↔ 避祸」确认为首要维度
- 决策游标从一维标量升级为可扩展多维向量
- 干系人画像直接影响价值光谱定位

### A.4 输入与交互

- 标题为唯一必填字段
- 环境上下文采用渐进积累模式（非一次性录入）
- 系统追问支持三个时机：创建时 / 专设区域 / 审视时

### A.5 日志验证

28 篇用户日志 (2025.12 ~ 2026.3) 验证了以下设计概念与用户自发描述同构：争得 vs 避祸光谱、三层架构、时间块日历、任务最小分割粒度、Kairos 多维度。
