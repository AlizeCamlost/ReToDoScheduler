# 实施教训

最后更新：2026-04-04

本文件记录用户纠错后沉淀下来的具体教训。`AGENTS.md` 只负责规定“何时读取、何时复检、何时回写教训”的机制；这里负责承载教训内容本身。

## 使用方式

1. 在最终确定 roadmap 或原子任务方案前，先扫描与当前改动区域相关的教训条目。
2. 生成方案时，把这些教训当成约束，而不是实现后再补救。
3. 正式改代码前，做一次 lesson review：逐条检查方案是否触犯这些教训；若触犯，先重做方案。
4. 实现完成后、提交前，再做一次 lesson review，检查旧问题是否被重新引入。
5. 当用户指出新的、可复用的教训时，在同一轮把它补到这里，并同步更新 `docs/README.md`。

## 教训条目

### L001 声明式优先，排除 GeometryReader 式补丁

- 适用范围：SwiftUI 容器、安全区、裁剪壳、dock 编排、滚动延伸。
- 反面模式：引入 `GeometryReader`、手算 reserve / inset、普通 `padding` / `offset` 去模仿本该由声明式容器关系表达的行为。
- 正向约束：优先使用 `safeAreaInset`、`safeAreaPadding`、`overlay`、`background`、容器职责拆分和原生状态驱动。

### L002 不要删除 safeAreaInset 语义，也不要用 padding 模仿安全区延伸

- 反面模式：删除承担真实语义的 `safeAreaInset`，或用普通 `padding` 去伪造安全区避让、dock 编排或沉浸式延伸。
- 正向约束：如果问题本质上是 safe-area 语义问题，就继续用 safe-area 语义修，而不是改成命令式布局补丁。

### L003 滚动裁剪范围必须保持沉浸式 edge-to-edge 延伸

- 反面模式：修一个容器问题时，把滚动裁剪范围的 top / bottom / horizontal edge-to-edge 延伸弄丢。
- 正向约束：凡是改 page shell、safe area 或 dock 结构，都要把滚动裁剪范围延伸作为显式检查项。

### L004 Sequence 输入框在 tab 右滑时必须能自然下沉

- 反面模式：sequence 到其他 tab 的切换过程中，输入框不再下沉，或者下沉语义和 tab/page 编排脱节。
- 正向约束：涉及 root shell、tab 切换、dock 承载方式的改动时，必须显式复检输入框下沉行为。

### L005 输入框下沉不能引起滚动偏移跳变

- 反面模式：dock 的出现、消失或下沉导致 sequence 滚动位置发生肉眼可见的跳变。
- 正向约束：一切 dock 编排调整都必须把“滚动偏移稳定性”作为单独检查项。

### L006 Sequence 重排不要回退到浏览态拖拽补丁

- 适用范围：`Sequence` 当前序列的编辑态、卡片重排、编辑态收口。
- 反面模式：在正常浏览态就挂拖拽交互，导致卡顿、误触和语义混乱；或者把整页 / 整行都变成手势主战场，破坏分页与滚动边界。
- 正向约束：浏览态只保留滚动和点按；必须先通过足够长的长按进入页面级编辑态，再启用拖拽重排和左滑动作。若原生列表交互无法同时满足“卡片外保留分页滚动、卡片内长按拖拽、卡片内左滑操作”这组约束，允许把实时测量与悬停偏移局限在当前序列卡片、且只在编辑态启用，不能回退到浏览态拖拽或整页手势补丁。

### L007 Sequence 编辑态只能改卡片手势，不要改页面手势边界

- 适用范围：`Sequence` 当前序列的编辑态、横向分页切换、纵向滚动和卡片手势。
- 反面模式：进入编辑态后，给整个列表或整页挂 tap / drag / swipe 手势，导致卡片外区域不再保留原来的横向翻页和纵向滚动，或者反过来把卡片内的拖拽重排与左滑动作抢掉。
- 正向约束：编辑态只允许当前序列卡片接管手势；卡片外区域继续保持浏览态的页面级滚动与翻页语义。卡片内通过长按拖拽与卡片内左滑操作托盘完成重排和操作；如果当前序列已经脱离 `List`，就不要再假定系统 `swipeActions` 还能可靠工作。退出编辑则依赖显式“完成编辑”或真正的失焦事件。

### L010 Sequence route B 不要再赌 `ScrollView` 里的系统 `swipeActions`

- 适用范围：`Sequence` 当前序列从 `List` 切到 `ScrollView / LazyVStack` 后的编辑态左滑操作。
- 反面模式：在 `ScrollView` 卡片行里反复微调 `highPriorityGesture`、`simultaneousGesture` 或其它手势优先级，试图把系统 `swipeActions` 调到可用。
- 正向约束：如果当前序列为了实现 route B 已经离开 `List`，左滑操作就直接做成卡片内自管 action tray；不要把问题误判成“滑得不够快”或单纯的手势优先级问题。

### L008 TaskPool 目录页必须保持导航与目的地的垂直语义

- 适用范围：`TaskPoolTreeBrowser` 的目录导航树、当前目录目的地和层级表达。
- 反面模式：把导航区和目的地区做成互不相干的摘要卡片，或者把目的地下的子目录继续按树缩进处理，导致层级关系和信息架构都变得含混。
- 正向约束：目录页应表达为上半屏导航、下半屏目的地的垂直 `nav / navdestination` 关系；两段都可以独立折叠。导航树用更明显的父子缩进表达层级，箭头贴近文件夹图标、数量右对齐；目的地下的子目录保持平铺列表，不继承树缩进。

### L009 同一原子操作的代码与文档必须同一笔提交

- 适用范围：任何会改变实现行为、交互语义、规格说明或 runbook 的原子 feature / fix 节点。
- 反面模式：先提交代码、再补一笔 docs；或者把同一个 atomic change 拆成“代码 commit”和“文档 commit”，导致 review 时看不到完整真相源。
- 正向约束：只要该原子操作需要同步代码和文档，就必须在同一个 commit 里一起提交。roadmap 节点要按原子性拆分，而不是按“代码 / 文档”拆分。

### L011 工程入口迁移后，脚本与入口文档必须同轮收口

- 适用范围：Xcode 工程路径、启动脚本、workspace package 命令、README、runbook、agent notes。
- 反面模式：iOS 客户端已经迁到新工程或新目录，但脚本、package 命令、根 `README.md`、`docs/runbooks/ios.md`、`apps/mobile/AGENTS.md` 仍残留旧入口，或者继续并列多个名义不同、行为相同的打开命令，导致项目记忆和真实工作流分叉。
- 正向约束：只要工程入口或源码根目录发生迁移，就必须在同一轮同步更新命令入口、仓库入口文档、相关 runbook 和 agent memory，并实际验证保留下来的唯一 helper 命令确实指向当前工程。若团队实际工作流是手动打开 Xcode 工程，文档应以 GUI 入口为主，CLI 只保留一个可选 helper。
