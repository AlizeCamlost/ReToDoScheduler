# 产品模型

相关文档：

- [architecture.md](architecture.md)
- [scheduling-model.md](scheduling-model.md)

## 1. 系统目标

ReToDoScheduler 不是静态 todo list，也不是一次性离线排程器。它长期维护一个任务池，并在当前状态下滚动派生调度视图。

系统的两个稳定目标只有两个：

- 任务入池本身完成认知卸载
- 在当前容量与观察窗口下回答“现在该怎么安排”

## 2. 稳定概念边界

概念层仍保留两层：

- `Norn`: 任务池维护层
- `Kairos`: 动态调度层

但在产品文案和代码实现中，统一使用直白实体：

- `Task`
- `TaskStep`
- `TaskGraph`
- `TimeTemplate`
- `ScheduleView`
- `TaskPoolOrganizationDocument`

任何新的产品观念，最终都应该落在这些稳定对象上，而不是再平行创造一套新的核心模型。

## 3. 核心抽象

- 系统长期维护的是任务池，不是静态日程表。
- 任务池除了 `Task` 集合，还包含一份可同步的组织文档，用来表达目录树和画布这两种浏览形态。
- 调度真正操作的是 `TaskStep`，不是抽象口号或任务分类。
- 依赖、子步骤序列、步骤进度、时间模板和观察窗口共同定义了当前可排的状态空间。
- 当前 `TaskStep` 默认按串行链推进；并行子任务仍保留为后续需求，通过 `concurrencyMode` 扩展，而不是在现阶段把双模交互混进主路径。
- “任务序列”当前只是 `Task` 的一层批量录入与轻分组语义：保存后仍落为多条 `Task`，而不是新增一类并行核心模型。
- 目录树与画布共用同一份 `TaskPoolOrganizationDocument`；它描述任务归属与可视化组织，不改变 `Task` 与 `TaskStep` 的调度语义。
- UI 中的原型、标签、提醒策略、AI 提示都只是输入层或交互层，不是调度本体。

## 4. 价值语义

从产品角度看，Kairos 可以被收束为：

- 硬约束
- 价值最大化

当前系统把价值持久化为一个二元组：

- `rewardOnTime`: 在 DDL 前完成的收益
- `penaltyMissed`: 错过 DDL 的损失

用于判断这组价值的主要产品语义包括：

- “争得 ↔ 避祸”这条主价值轴
- 复利收益，如技能、资历、关系、声誉、影响力
- 现实损失，如违约、事故、阻塞、错失窗口

这些因素可以持续扩展，但扩展方式应保持克制：

- 如果它决定可不可排，它属于硬约束
- 如果它决定值不值得排，它属于价值评估或比较器输入
- 不应把新因素升格为第三类核心调度要素

## 5. 输入契约

最小输入要求：

- 标题必填

建议进入系统的稳定字段：

- `dueAt`
- `estimatedMinutes`
- `minChunkMinutes`
- `dependsOnTaskIds`
- `steps`
- `steps[].progress`
- `scheduleValue`
- 自然语言备注

AI 可以帮助提取、补全和建议这些字段，但产品基础能力不依赖 AI。

补充约束：

- Quick Add 可以一次录入一组任务序列，但它们保存后仍是普通 `Task`；共享的 bundle 元数据只服务于录入便利和界面上的轻分组展示。
- 目录树与画布的组织信息必须进入同步边界；未实现该能力的客户端可以不提供 UI，但不能在同步时把这份文档清空。

## 6. 用户得到什么

当前产品最重要的输出是：

- 认知卸载后的任务池
- 可在目录树 / 画布间切换且可跨端保真的任务池组织视图
- 基于当前窗口的任务序列
- 可承诺的时间块安排
- 跌落后仍可从“现在”继续接上的滚动调度

## 7. 非目标

- 不把所有哲学判断都直接编码成一等 schema
- 不把 UI 分类法误当成调度真相源
- 不把 AI 的提示词结构误当成领域模型
