## 1. 愿景

一个**会自我进化的外脑系统**，通过三层协作帮助用户管理待办事项：

| 层 | 职责 | 本质 |
|---|------|------|
| **确定性规则引擎** | 可预测、可审计、可导出的调度逻辑 | 用户信任的、看得见摸得着的纯文本规则 |
| **AI 估算层** | 从自然语言和历史行为中提炼时间/拆分/优先级建议 | 外部 API，提供建议但不做最终决定 |
| **人类反馈环** | 用户的调整被前两层吸收，系统自我改造 | 让系统越用越懂用户 |

核心飞轮：**人 → AI 建议 → 人调整 → 规则沉淀 → 更好的 AI 建议 → …**

### 1.1 「争得 — 避祸」价值光谱

所有任务分布在一条价值光谱上，由两极定义：

```
争得 ←————————————————————————————→ 避祸

"不做死不了，做了开启可能性"          "做了不改变现状，不做带来毁灭性后果"
 高收益 · 雪球效应 · 机会驱动           防御性 · 不可接受损失 · 存续驱动
```

- 每个任务在此光谱上有一个**固有位置**（Phase α 手动标注；Phase β+ AI 从自然语言推断；Phase γ+ AI 从历史模式自动定位）
- 用户有一个**决策向量**（初始为一维标量「争得↔避祸」，随 Norn 成熟逐步扩展为多维），随时间/境遇/心境动态变化
- Kairos 依据任务位置 × 决策向量来加权优先级排序
- 决策向量的维度是**可扩展的键值对结构**，新维度通过使用数据浮现，AI 提议 → 用户命名确认后加入
- 从用户日志中已浮现的候选维度：
  1. 争得 ↔ 避祸（已确认为首要维度）
  2. 干系人权重（来源人的重要性）
  3. DDL 紧急度
  4. 个人成长价值
  5. 难度/耗时/风险
  6. 曝光/影响力（在关键人面前的可见性）

## 2. 核心概念

### 2.1 Norn — 内容组织模型

> Norn（命运织者）：定义待办事项如何被记录、结构化、分类和关联。

#### 2.1.1 任务原型分类（草案 v0.2）

六种原型适用于生活和工作两个领域，工作任务在同一套原型上叠加更丰富的元数据：

| 原型 | 时间行为 | 调度决策权重 | 生活例子 | 工作例子 |
|------|---------|-------------|---------|---------|
| **时限义务** | 休眠 → 唤醒 | 中~高 | 换驾照 | 写ToD文档(DDL一周)、提API申请(DDL半周) |
| **随缘备忘** | 温度递升 → 审视 | 低 | 改与会人身份 | review同事代码、编译效率优化(良心活) |
| **周期仪式** | 周期触发 | 低 | 每周审视 | 逆向改进月报 |
| **修行课题** | 持续在场 + 检查点 | 高 | 品味工作日志 | 责任田交接(多步骤)、safeArea议题梳理 |
| **轻量种草** | 永久休眠 | 无 | 学金融 | — |
| **Techne（技艺）** | 持续在场 | 中→高 | 歌单同步工具 | 整理自测试规范(团队贡献+技能) |

**分类轴：**
- **时间行为**：休眠→唤醒 / 永久休眠 / 周期触发 / 持续在场
- **调度决策权重**：高（竞争有限时间）/ 中 / 低（固定时间）/ 无（被动队列）
- **领域**：生活 / 工作（影响可调度时间域）

#### 2.1.1b 工作任务元数据层

工作任务在原型之上需要额外元数据（生活任务可选填）：

| 元数据 | 说明 | 输入方式 |
|--------|------|---------|
| **干系人** | 谁关心这个任务，关系动力学 | 从环境上下文选择（见 2.1.3） |
| **阻塞关系** | 你阻塞谁 / 谁阻塞你 | 手动标注 |
| **绩效挂钩** | 影响哪个考评维度 | 从环境上下文选择 |
| **耗时范围** | 支持乐观/悲观双值（如 0.5~2天） | 手动或 AI 推断 |
| **信息不确定性** | 耗时是否取决于调查结果（条件分支） | 手动标记 |

#### 2.1.2 关于周期仪式的设计决策（待定）

初步建议：
- **审视仪式**（每周/每月审视）→ 内建为 Kairos 节拍器机制，不作为 Norn 任务
- **周期性事务**（月度结算、月报）→ 交给外部提醒工具（iOS 提醒事项），不纳入 Norn 任务池
- **工作周期仪式**（逆向改进月报等）→ 待确认归属

*（待用户确认）*

#### 2.1.3 干系人知识图谱（渐进式构建）

不做一次性录入，从日常任务输入中逐步积累：

```
原始层（自动积累，从任务描述中提取的文本片段）:
  "刘某，同组，职级略高，有求于他"
  "大SE，必须留好印象、建立影响力"
  "外包，低半级，坏印象会传导到其组正式员工"

结构化层（AI 渐进提炼，用户确认后生效）:
  刘某:
    关系: 同组同事
    职级: 略高
    动力学: 需配合、留好印象
    相关任务优先级影响: 中等加成
  大SE:
    影响力: 高
    相关任务优先级影响: 显著加成（即使任务本身价值低）

主动补问（AI 驱动，系统识别知识缺口后择机提问）:
  时机：创建任务后 / 专设「系统想了解」区域 / 审视仪式时
  用户可回答可跳过，无强制
```

干系人画像直接影响任务在高维价值光谱上的定位。

#### 2.1.4 工作流模板（Workflow Template）

需求开发等有固定阶段的重复性工作，用**模板**承载：

```
模板: 需求开发（1~2 周）
  ① 设计文档
  ② 测试串讲
  ③ 开发
  ④ 自验证迭代
  ⑤ 质量测试
  ⑥ 上库流程（~1 周）
```

**关键简化**：工作流不需要独立 UI，复用修行课题的进展队列结构即可。用户只关心最新更新状态，工作流中的当前阶段在任务列表中与普通任务同级呈现。

- Phase α：修行课题的进展队列同时承载工作流跟踪
- Phase β+：模板实例化，新需求一键创建含预填阶段的进展队列

#### 2.1.5 最简输入规范 + 动态提示

| 字段 | Phase α（无AI） | Phase β+（有AI） |
|------|----------------|-----------------|
| 标题 | **必需** | **必需** |
| 原型 | 必需（下拉选择） | AI 推断 → 人确认 |
| 截止日 | 有则填 | AI 从文本提取 |
| 耗时估算 | 有则填（支持乐观/悲观范围值） | AI 推断 |
| 干系人 | 有则选 | AI 从文本匹配知识图谱 |
| 自然语言备注 | 可选 | 越多越好，供 AI 学习 |

**唯一不可缺失：标题。** DDL 如果存在但没填，后果由用户承担。

**动态输入提示**（Phase β+）：系统根据上下文智能追问关键信息，可跳过。

有了知识图谱后，最简输入示例：
> `提API申请，找刘某，本周` → 系统自动补全干系人关系、推断原型和价值位置

### 2.2 Kairos — 调度与优先级算法

> Kairos（恰当时机）：定义系统如何帮用户决定「现在该做什么」。

#### 2.2.1 三层架构（草案 v0.3）

AI 与确定性规则的分工：AI 做创造性判断（调度主力），确定性规则做刚性校验（兜底保障）。

```
┌─────────────────────────────────────────────┐
│            人类反馈环                         │
│  用户对 AI 建议的调整                         │
│  → AI 自动发现统计规律，以规则建议形式呈现     │
│  → 用户审核通过后纳入确定性规则               │
└────────────────┬────────────────────────────┘
                 ↓ 吸收
┌─────────────────────────────────────────────┐
│     AI 调度层（外部 API，启发式，调度主力）    │
│  · 自然语言 → 结构化字段（截止日/风险/耗时）   │
│  · 任务拆分策略                              │
│  · 优先级排序                                │
│  · 时间片分配                                │
│  · 历史行为模式 → 校准                       │
│  · 触发时机：                                │
│    - 任务创建/编辑时（解析输入）              │
│    - 队列变动时（任务完成/新增 → 重排）       │
└────────────────┬────────────────────────────┘
                 ↓ AI 输出
┌─────────────────────────────────────────────┐
│    确定性校验层（纯文本可导出，刚性兜底）      │
│  · 不遗漏：所有应调度任务均已分配时间片        │
│  · 不差错：分割后耗时总和 = 原始估算           │
│  · DDL 守护：有截止日的任务在 DDL 前有足够时间  │
│  · 排序校验：结果符合设计约束（如大块优先）    │
│  · 原型规则：可见性行为（唤醒/温度递升/…）     │
│  · 学习到的校准参数                           │
└─────────────────────────────────────────────┘
```

**Phase α（无 AI）**：用户手动拖拽排列时间块，确定性校验层提供基础保障（不遗漏、DDL 预警）。调度能力有限，用户接受。核心价值在 Norn（捕获组织）+ 数据采集。

#### 2.2.1b 任务可分割度（调度硬约束）

任务有**最小不可分割粒度**，这是 Kairos 的装箱约束：

| 可分割度 | 最小块 | 调度特征 | 例子 |
|---------|--------|---------|------|
| 极细 | ~10 min | 随时做，填缝隙 | 回消息、看一篇文章 |
| 中等 | ~1 hr | 需要连续时间段 | review 代码、月报 |
| 粗 | ~2-4 hr | 需要半天整块 | 解 bug 走读、窗口对线 |
| 不可分割 | 半天+ | 必须完整大块 | 需求开发-开发阶段、串讲 |

**调度策略**：大块任务需要**优先级加成**——可用的大时间窗口稀缺，如果不优先安排，会被碎片任务不断挤走。本质是装箱问题：大块先放。

#### 2.2.1b Kairos 输出形式

- **主视图**：时间块日历（日视图 + 周视图），任务占据具体时间片
- **辅助视图**：任务序列总表（全量任务按优先级排列的队列视图）

#### 2.2.1c 工作/生活双时间域

工作任务和生活任务通常调度在不同时间域，紧急时允许互相侵占：

```
工作时间域: [工作日 工作时段]  ← 主要排工作任务
生活时间域: [下班后 + 周末]    ← 主要排生活任务
溢出规则: 紧急工作DDL可侵占生活时间（反之亦然）
```

#### 2.2.1d 条件分支耗时

工作任务常见「耗时取决于调查结论」的模式，系统支持：
- **乐观估算**：最好情况的耗时
- **悲观估算**：最坏情况的耗时
- **当前判断**：待调查 / 偏乐观 / 偏悲观
- 调度时按悲观估算预留时间，实际进展后可修正

#### 2.2.2 随缘备忘「温度递升」策略

| 阶段 | 入库后时间 | 系统行为 |
|------|-----------|---------|
| 冷藏 | 0~7 天 | 完全隐藏，脑卸载成功 |
| 微温 | 7~30 天 | 每周在「轻推」区域出现 1 次 |
| 升温 | 30~90 天 | 每周出现 2~3 次，视觉权重提高 |
| 审视 | 90 天+ | 触发「这件事还有意义吗？」三选一：做掉 / 重置冷藏 / 无愧归档 |

阶段参数作为确定性规则，可导出可调。

#### 2.2.3 时限义务 AI 解析流程

1. 用户输入自然语言描述（如「换驾照，6月到期，要去车管所」）
2. AI 解析输出：截止日、外部依赖、风险因子、建议唤醒日、建议执行窗口
3. 用户确认或修正
4. 修正结果回馈 AI 上下文，校准未来解析
5. 解析用的 prompt 模板本身为纯文本，可导出

#### 2.2.4 修行课题进展结构

```
课题: "<标题>"
总目标: "<描述>"
最终期限: <日期>
进展队列:
  - 日期: <更新日期>
    内容: "<进展描述>"
    下次检查点: <日期>
最后更新: <日期>
下次提醒: <日期>
```

#### 2.2.5 Techne（技艺）双态模型

> 命名来源：希腊语 τέχνη，亚里士多德定义的「能产出成果的实践知识」。对应道家内丹（技能修炼 / Techne of Self）与外丹（造物 / Techne of Craft）。

- **轻量态**：单条任务，AI 估算耗时，简单则直接完成归档
- **升级操作**：用户或 AI 判断为复杂 → 转化为修行课题，获得子项/进展跟踪能力

#### 2.2.6 精力水平因子

**Phase α 策略**：视为恒定，不作为调度输入。但系统从 Phase α 起即收集以下执行力数据，为未来精力建模埋点：

| 采集指标 | 含义 | 未来用途 |
|----------|------|---------|
| 完成度 | 任务是否完成、部分完成 | 估算任务实际难度 vs 预估难度 |
| 实际耗时 vs 预估耗时 | 用户的估算偏差 | 校准耗时估算系数 |
| DDL 遵守度 | 任务是否在截止日前完成 | 识别哪类任务容易拖延 |

**未来 Phase**：基于积累数据建立精力波动模型，使调度考虑「什么时候适合做什么」。

#### 2.2.7 渐进自主化迭代路线图

系统从「笨但可靠」演进到「聪明且自适应」，每个 AI 组件都有手动回退路径：

| 能力 | Phase α（基石） | Phase β（AI 辅助） | Phase γ（AI 推断） | Phase δ（AI 自主） |
|------|----------------|-------------------|--------------------|-------------------|
| 任务输入 | 手动填表单字段 | AI 解析自然语言 → 人确认 | AI 推断隐含字段 → 人微调 | 用户只写一句话 |
| 价值光谱 | 手动设定位置 | AI 从描述建议 → 人确认 | AI 从历史模式推断 | AI 自动定位 |
| 决策向量 | 一维标量（争得↔避祸） | 探索性新维度出现 | 多维向量成形，AI 辅助诠释 | 向量自校准 |
| 时间可用性 | 手动粗略周模板 | 手动 + 微调 | 外部日历导入 | 邮件/日历自动同步 |
| 耗时估算 | 手动输入 | AI 建议 + 人确认 | AI 估算 + 校准系数 | 自动估算 |
| 精力水平 | 视为恒定；采集执行力数据埋点 | 初步模型，时段偏好 | 精力波动建模 | 自适应精力感知 |
| 规则沉淀 | 纯手动配置 | AI 发现模式 → 规则建议 → 人审核 | 半自动沉淀 | 自适应 |

**关键原则**：Phase α 是完整可用的产品，不依赖任何 AI。后续叠加不破坏已有功能。

#### 2.2.8 处理管道架构

```
任务输入管道:
  [用户输入]
    → [解析器]   α: 表单直填  β+: AI 自然语言解析
    → [增强器]   α: 无操作    γ+: AI 推断隐含字段
    → [用户审阅] ← 永远保留
    → [存入 Norn]

调度管道:
  [Norn 任务] + [时间可用性] + [决策向量] + [精力水平]
    → [优先级评分器]  α: 确定性公式  γ+: 公式+AI加权
    → [时间片分配器]
    → [日历视图(日/周)] + [任务序列总表]

反馈管道:
  [用户调整记录]
    → [模式检测器]  α: 禁用  β+: AI 分析
    → [规则建议]
    → [用户审核] ← 永远保留
    → [确定性规则更新]
```

## 4. 用户场景

### 场景 1：快速捕获（认知卸载）⏳ 远期
- **触发**：用户正在做任务 A，突然想到任务 B
- **操作**：用最快方式（悬浮按钮/语音）输入一句话
- **期望**：任务 B 入库，大脑释放「威胁」感，不打断任务 A 的心流
- **来源**：*「想到别的事，只做一件事：写在旁边清单，不立刻切换」（2.25 工作日志）*
- **Phase α 替代**：Web 端常规表单创建

### 场景 2：每日启动简报 ⏳ 远期
- **触发**：用户早上打开 app
- **操作**：查看 Kairos 生成的日计划
- **期望**：看到今天应该做的时间块安排，快速进入执行状态，抵御启动困难
- **来源**：*「调教AI，它能帮我排序。甚至可以每天早报晚报」（2.11 工作日志）*
- **Phase α 替代**：用户手动查看日/周视图

### 场景 3：承诺时间块 🔜 Phase α
- **触发**：用户决定开始工作
- **操作**：在日视图上拖拽任务到时间块
- **期望**：用户承诺的是时间而非结果，减轻完美主义焦虑
- **来源**：*「承诺时间块，而不是承诺任务完成」（2.25 工作日志）*

### 场景 4：跌落后回归 ⏳ 远期
- **触发**：用户几天没使用系统（精力低谷/情绪波动）
- **操作**：重新打开 app
- **期望**：系统不追责、不堆积罪恶感，而是提供「现在你可以从这里开始」的简洁切入点
- **来源**：*「一旦建立它将稳固运转，只要有一鼓作气的机会，我就能跃回这个稳态」（2.19 日省）*

### 场景 5：审视仪式（远视校准）⏳ 远期

### 第 1 轮：任务原型识别

用户提供 9 个实际待办样本，识别出 6 种原型：时限义务、随缘备忘、周期仪式、修行课题、轻量种草、炼器点子。确定分类轴为「时间行为」×「调度决策权重」。

### 第 2 轮：原型机制深化 + 系统哲学浮现

关键决策：
- Q1 → 时限义务用 AI 从自然语言解析唤醒参数，用户修正反馈校准
- Q2 → 随缘备忘采用「温度递升」策略，外化良心轮询
- Q3 → 修行课题为增量日志队列，带检查点提醒
- Q4 → 轻量种草（利己投资）vs 随缘备忘（社交信用）
- Q5 → 炼器点子支持升级为修行课题
- Q6 → 确认精力为调度因子

系统哲学浮现：三层架构（确定性规则引擎 + AI 估算层 + 人类反馈环），核心飞轮为「人→AI建议→人调整→规则沉淀」。用户明确要求确定性规则和 AI 共同估算耗时/拆分/调度，人工调整后被两层吸收自我改造。

### 第 3 轮：价值光谱 + Kairos 具体化 + 命名

关键决策：
- 提出「争得↔避祸」价值光谱：所有任务有固有位置，用户有动态游标，Kairos 据此加权
- Q7 → 反馈吸收：AI 自动发现统计规律 → 规则建议 → 用户审核 → 纳入确定性规则
- Q9 → AI 触发时机：创建/编辑时 + 队列变动时（任务完成/新增触发重排）
- Q10 → Kairos 输出：时间块日历（日/周视图）+ 任务序列总表
- 「炼器点子」更名为 **Techne（技艺）**，对应道家内丹/外丹，源自亚里士多德哲学

### 第 4 轮：渐进自主化架构 + 决策向量多维化

关键决策：
- Q11 → 价值光谱位置输入随迭代演进：手动→AI建议→AI推断→AI自动
- Q12 → 决策游标升级为**可扩展多维向量**，初始一维（争得↔避祸），新维度随使用浮现
- Q13 → 时间可用性暂用手动粗略周模板，未来接入外部日历/邮件
- Q14 → 延后到 UI 讨论

架构决策：
- 提出**渐进自主化四阶段路线图**（α基石 → β辅助 → γ推断 → δ自主）

- Phase α 是完整可用产品，不依赖 AI，可立即建设
- 提出三条处理管道架构：任务输入管道、调度管道、反馈管道
