# AI 协作 UI 开发：决策先行方法论

## 问题

在传统开发中，写代码本身是一种思维外化工具——你实现一个交互，发现它不对，你就知道自己要什么了。代码是逼出决策的过程，而不仅仅是决策的产出。

AI 打破了这个机制。它直接生产完整代码，跳过了你本该在实现过程中被迫做出的那些微观决策。结果是：代码存在了，但决策没有发生。你对着一个"完整"的页面感到不满意，却无法有条理地挑剔它——因为你从未经历过那个让你逐步清晰自己要什么的过程。

## 方法

### 一、用第一人称叙事替代设计讨论

不要试图描述"设计"。为你要做的屏幕写一段第一人称旁白：

> 打开 app 的首屏，我看到屏幕的主体部分是一个沉浸式 edge-to-edge 的滚动视图，底部衬着一个 safeAreaInset 装载的悬浮岛胶囊式的输入框。滚动视图内依次是……

注意这段叙述的特征：

- **使用精确的技术词汇**（`safeAreaInset`、`edge-to-edge`），而不是模糊的感受词（"简洁的"、"现代感的"）
- **描述空间关系和顺序**（"依次是"、"底部衬着"），而不是孤立地描述单个组件
- **对不确定的部分保持沉默**，而不是用模糊语言填充

这段叙事写不下去的地方，精确指向你还没做完的决策。

### 二、要求 AI 枚举未决问题

叙事写完后，不要让 AI 直接开始实现，而是：

> 请在此基础上将动效、状态管理、交互边界等问题充分考虑，专注最大的 UX 框架，暂时忽略每个版块的细化，告诉我还有哪里没细化，我补充。

AI 应该输出一份编号的问题清单。比如：

1. 三屏之间的切换，点按钮和手势滑动的动画是否一致？
2. 悬浮岛消失是跟随手势实时变化，还是松手后触发？
3. 滚动时顶部 header 是固定还是跟随滚动？

这一步的价值在于：让 AI 帮你找到你还没意识到的决策点。

### 三、用约束声明替代需求规格

对 AI 最有效的指导不是描述你想要的结果，而是声明实现约束：

> 所有动效尽可能采用原生的属性事件和状态管理组合控制，而非自定义回调硬编码，因为后者会引入复杂的时序处理。

对比无效表述：

> 动画要流畅自然。

约束声明直接限制了 AI 的实现空间，而结果描述让 AI 自由发挥——自由发挥正是产生你不满意的代码的原因。

### 四、用排除法做决策

这次对话中最高效的决策方式不是"我要 X"，而是"我不要 Y"：

> 顶部的 header 我根本不需要，这是某个过去遗留的随意性实现产生的，帮我根除它。

排除法有两个优势：

- 减少 AI 需要处理的复杂度
- 避免你在不确定要什么的时候被迫给出一个正面答案

当你不确定要什么时，你几乎总是确定不要什么。从这里开始。

### 五、要求完整确认稿，确认后才开始

所有决策做完后：

> 请再明确一次完整的 UX 和文件结构，由我确认后开始开发。

确认稿必须是可实现的规格，不是概念描述。它应该包含：

- 导航结构（层级、切换方式、手势方向）
- 各屏结构（固定区域 vs 滚动区域、内容顺序）
- 状态驱动关系（什么状态控制什么行为）
- 文件目录

不确认不写代码。

### 六、在架构选择上要求对比分析

当面对多种实现方式时，不要让 AI 直接选一种，而是：

> 这两种目录组织方式，请从未来 feature 扩展和架构演进的前瞻角度给我输出比较和建议。

这迫使 AI 输出决策依据而非决策结论，让你基于理由做判断。

### 七、命名和术语：大巧不工

对 AI 生成的命名保持警惕。AI 倾向于生产带有"产品感"的造词（TaskFeed、TaskVault），而工程命名应该准确描述事物本身。

判断标准：一个新加入项目的人看到这个名字，是否能不查文档就知道它是什么。

## 决策顺序

```
1. 写第一人称叙事 → 发现哪些决策还没做
2. AI 枚举未决问题 → 暴露盲区
3. 逐条回答（能排除的就排除） → 完成决策
4. AI 输出完整确认稿 → 确保理解一致
5. 确认 → 开始实现
6. 实现后只处理 A 类（视觉感知）问题 → 迭代收敛
```

## 两类问题的分类法

在任何 UI 讨论中，随时用这个分类检查自己卡在哪里：

- **A 类（视觉/感知）**：间距、颜色、动画节奏、层次感。只能看到了才能判断，属于实现后迭代的范畴。
- **B 类（交互/结构）**：什么放在哪、层级关系、状态驱动逻辑、边界行为。这是逻辑问题，必须在写代码之前解决。

如果你面对一个页面感到"哪里不对但说不清"——你卡在的几乎一定是 B 类问题。回到第一步。
