# 产品模型

相关文档：

## 1. 系统目标

ReToDoScheduler 的产品目标是把“任务记录”和“任务调度”拆成两个稳定协作层：

- 任务池：负责记录、结构化和维护任务
- 调度器：负责在当前状态下派生可执行安排

系统不是单纯的 todo list，也不是单次离线排程器；它是一个持续生长的任务池与滚动调度系统。

## 2. 设计原则

- 录入本身即价值：即使暂时不调度，任务入池也能完成认知卸载
- 当前状态优先：调度基于“现在的任务池 + 现在的容量 + 当前观察窗口”重算
- 纯手动可用：核心产品必须不依赖 AI 也能成立
- AI 只做增强：未来 AI 可以帮助结构化、估时、排序和规则沉淀，但不应破坏手动可控性
- 纯文本可导出：用户数据和规则最终应可被审计和迁移

## 3. 概念边界

概念层仍使用：

- `Norn`: 任务池维护层
- `Kairos`: 动态调度层

但在实现与产品文案上，统一采用直白命名：

- `Task`
- `TaskStep`
- `TaskGraph`
- `TimeTemplate`
- `ScheduleView`

## 4. 价值体系

### 4.1 价值光谱

所有任务分布在一条“争得 ↔ 避祸”的价值光谱上：

```text
争得 ←————————————————————————————→ 避祸

做了开启可能性                      不做会带来现实损失
收益、积累、成长                      存续、防御、止损
```

这是产品层的首要价值轴。

### 4.2 决策向量

用户对任务的取舍不是单一优先级，而是一个可扩展决策向量。当前已识别的候选维度包括：

- 争得 ↔ 避祸
- 干系人权重
- DDL 紧急度
- 个人成长价值
- 难度 / 耗时 / 风险
- 曝光 / 影响力
- 可分粒度 / 可并行度

调度实现不必一次做完所有维度，但产品模型应允许它们逐步进入系统。

## 5. 任务原型

当前保留六类任务原型：

| 原型 | 时间行为 | 典型特点 |
|---|---|---|
| 时限义务 | 休眠 → 唤醒 | 有明确 DDL，接近截止前竞争力强 |
| 随缘备忘 | 温度递升 → 审视 | 不必立刻做，但会逐渐重新进入视野 |
| 周期仪式 | 周期触发 | 定期发生的复盘或事务 |
| 修行课题 | 持续在场 + 检查点 | 长期推进，有进展记录 |
| 轻量种草 | 永久休眠 | 低强度兴趣项，不主动抢占当前容量 |
| Techne | 持续在场 | 技能修炼与造物混合型任务 |

分类轴保留三条：

- 时间行为
- 调度决策权重
- 领域（生活 / 工作）

## 6. 工作任务扩展信息

工作任务在原型之上通常还需要：

- 干系人
- 阻塞关系
- 绩效挂钩
- 乐观 / 悲观耗时
- 信息不确定性

这些字段不一定都在首轮实现中落地为一等类型，但在产品模型中应保留位置。

## 7. 干系人与工作流

### 7.1 干系人知识图谱

干系人信息采用渐进式积累：

- 原始层：从任务描述中积累文本片段
- 结构化层：逐步提炼为稳定关系字段
- 主动补问：在创建或审视时补齐关键缺失信息

### 7.2 工作流模板

需求开发等重复性流程，应使用模板承载阶段结构。工作流不必一开始就有独立 UI，但应允许被复用为任务内部步骤序列或进展结构。

## 8. 输入规范

最小输入要求：

- 标题必填

其余字段根据阶段逐步增加：

- 截止日
- 耗时估算
- 原型
- 干系人
- 自然语言备注

后续 AI 增强可从自然语言里提取和补全这些信息，但产品基础能力不依赖 AI。

## 9. 用户场景

当前最重要的用户场景：

- 认知卸载：快速把任务放进任务池
- 查看调度：根据当前窗口查看任务序列与时间块
- 承诺时间块：承诺投入时间，而不是空喊结果
- 跌落后回归：几天不用后仍能从“现在”重新接上
- 审视仪式：在执行与战略校准之间切换
